## 2. Các nhóm chỉ số (10 categories)

### 2.1 Volume — đếm bao nhiêu

| Metric | Định nghĩa | Đơn vị | Lumi |
|---|---|---|---|
| **Total tokens** | Sum input + output + cache | tokens | ✅ |
| **Input/prompt tokens** | Tokens sent to model | tokens | ✅ |
| **Output/completion tokens** | Tokens generated by model | tokens | ✅ |
| **Total requests / turns** | # API calls | count | ✅ |
| **Total sessions** | # distinct sessions | count | ✅ |
| **Total active users** | # distinct users (in period) | count | ✅ |
| **Avg tokens / request** | total_tokens / requests | tokens | ⚠️ derive được |
| **Avg tokens / session** | total_tokens / sessions | tokens | ⚠️ derive được |
| **Avg tokens / user** | total_tokens / users | tokens | ⚠️ derive được |

> **Ý nghĩa:** "Hệ thống đang tiêu thụ bao nhiêu." Đây là base layer — phải có hết.

### 2.2 Cost — tốn bao nhiêu tiền

| Metric | Định nghĩa | Đơn vị | Lumi |
|---|---|---|---|
| **Total cost** | Sum input + output + cache cost | USD | ✅ |
| **Input cost** | Cost của input tokens | USD | ⚠️ có raw, chưa split rõ ở KPI |
| **Output cost** | Cost của output tokens | USD | ⚠️ có raw, chưa split rõ ở KPI |
| **Input vs Output cost ratio** | input_cost / total_cost | % | ❌ |
| **Cost per request** | total_cost / requests | USD/req | ✅ ("Avg Cost / Turn") |
| **Cost per session** | total_cost / sessions | USD/session | ❌ chưa canonical |
| **Cost per user** | total_cost / users | USD/user | ⚠️ ("Avg / User" ở Overview) |
| **Cost per 1000 tokens** | total_cost / total_tokens × 1000 | USD/1K | ❌ |
| **Cost per 1000 requests** | total_cost / requests × 1000 | USD/1K req | ❌ (normalized comparison) |

> **Ý nghĩa kinh tế:**
> - **Cost per request/session/user** là 3 unit economics chuẩn FinOps. Cost per session đặc biệt quan trọng vì 1 session = 1 "đơn vị giá trị" (1 cuộc trò chuyện).
> - **Cost per 1000 tokens** normalize cho phép so sánh tháng này vs tháng trước kể cả khi volume thay đổi.
> - **Input vs Output ratio**: output đắt gấp 3-30× input. Nếu output >> input → model "lan man" → cải thiện prompt có thể giảm 50% chi phí.

### 2.4 Cache metrics — tiết kiệm bao nhiêu nhờ cache

| Metric | Định nghĩa | Lumi |
|---|---|---|
| **Cache write tokens** | Tokens written to cache (Anthropic: 1.25× input price) | ⏳ type có, BE pending |
| **Cache read/hit tokens** | Tokens read from cache (Anthropic: 0.1× input price) | ⏳ type có, BE pending |
| **Cache read ratio %** | hit / (hit + miss + write) — metric chính của Anthropic Dashboard | ❌ |
| **Cache hit rate over time** | Trend chart | ❌ |
| **Cache write cost** | $ trả cho write | ❌ |
| **Cache read cost** | $ trả cho read (cheaper) | ❌ |
| **Cache saving USD** | (input_price − cache_read_price) × hit_tokens | ⏳ UI placeholder "$0.00 BE pending" |
| **Write amortization** | cumulative read_tokens / write_tokens (Anthropic example: 8.65×) | ❌ |
| **Cache composition** | Stacked: write / read / no-cache tokens | ❌ |

> **Ý nghĩa kinh tế:**
> - Anthropic dashboard chính `platform.claude.com/usage/cache` chỉ có 3 metric: **read ratio**, **composition**, **write amortization**. Pattern Lumi nên copy.
> - **Write amortization 8.65×** = mỗi $1 trả cho cache write, sau đó tiết kiệm $8.65 nhờ cache reads. ROI rất cao nếu prompt template được reuse.
> - **Cache saving** = số tiền nếu KHÔNG dùng cache. Số này quan trọng nhất với CFO — "nhờ cache, công ty tiết kiệm $X tháng này".

### 2.5 Trend & Time-series — biến động theo thời gian

| Chart | Granularity | Lumi |
|---|---|---|
| **Daily cost trend** | per day | ✅ TokenLineChart |
| **Daily tokens trend** | per day | ✅ |
| **Hourly cost (last 24h/7d)** | per hour | ❌ anh đã yêu cầu |
| **Weekly comparison (WoW)** | week-over-week | ⚠️ có vs prev period chung |
| **Monthly comparison (MoM)** | month-over-month | ❌ |
| **YoY (year-over-year)** | yearly seasonality | ❌ |
| **Cumulative cost (current month)** | running total | ❌ |
| **Projected cost (end of month)** | linear extrapolation | ❌ |

> **Ý nghĩa kinh tế:**
> - **Cumulative + projected EOM** là cặp metric quan trọng nhất cho budget tracking. Cho phép "nếu cứ tốc độ này, cuối tháng sẽ tốn $X".
> - **Hourly breakdown** giúp phát hiện pattern (peak buổi tối? batch lúc 3am?). Useful để biết khi nào abuse xảy ra.

### 2.6 Distribution — phân bổ ai tốn nhiều

| Metric | Định nghĩa | Lumi |
|---|---|---|
| **Top N users by cost** | Ranking | ✅ Overview top 10 |
| **Top N users by tokens** | Ranking | ✅ |
| **Top N sessions by cost** | Ranking | ❌ |
| **p50/p90/p95/p99 cost per user** | Distribution percentiles | ❌ |
| **Mean vs median cost per user** | Skewness indicator | ❌ |
| **Cost concentration (Gini / top 10%)** | % cost do top 10% user gây ra | ❌ |
| **Heavy hitters list** | Users > X× median | ⚠️ logic có (`computeAlertThreshold`), chưa surface |

> **Ý nghĩa kinh tế:**
> - **Median vs mean**: nếu mean >> median → có "outlier". Ví dụ mean $10/user nhưng median $1/user → 1 user tốn $1000, 99 user tốn $1 mỗi người.
> - **Gini coefficient / top 10% share**: nếu top 10% user gây 90% cost → "long tail" mỏng → cân nhắc tier hoá user.
> - **Heavy hitters**: list cụ thể những user vượt threshold, kèm action (cảnh báo / block / upgrade tier).

### 2.7 Budget & forecast — quản lý ngân sách

| Metric | Định nghĩa | Lumi |
|---|---|---|
| **Monthly budget ceiling** | $ ngưỡng tháng | ❌ |
| **Spent vs budget %** | spent / budget | ❌ |
| **Days remaining in cycle** | đếm ngược | ❌ |
| **Avg daily spend (current month)** | spent / days_passed | ❌ |
| **Projected end-of-month cost** | avg_daily × days_in_month | ❌ |
| **Burn rate vs target** | actual_daily / (budget / days) | ❌ |
| **Days until budget exhausted** | budget_remaining / avg_daily | ❌ |
| **% of users having budget** | budget per user use case | ❌ |
| **Per-user budget** | $X/tháng cho user (FinOps standard) | ❌ |

> **Ý nghĩa kinh tế:**
> - Đây là layer mà CFO/CEO quan tâm nhất. **Projected EOM + days until exhausted** trả lời "tháng này có vượt budget không, hành động trong bao lâu nữa".
> - **Per-user budget** = guardrail. Set $X/user/tháng → user vượt thì throttle/email/upgrade tier.

### 2.8 Throughput / rate — tốc độ tiêu thụ

| Metric | Định nghĩa | Lumi |
|---|---|---|
| **Tokens per minute (TPM) — current** | Rolling avg last 5/15 min | ❌ (anh gọi "B-red") |
| **TPM peak (last 24h)** | Max trong 24h | ❌ |
| **Requests per minute (RPM)** | Rolling | ❌ |
| **Tokens per day** | Daily aggregate | ✅ |
| **% of TPM quota used** | current_TPM / API_rate_limit | ❌ |

> **Ý nghĩa kinh tế:**
> - **TPM/RPM rolling**: phát hiện "burst" — ví dụ retry loop, bot spam, abuse.
> - **% quota used**: capacity planning. Nếu peak TPM = 80% Anthropic rate limit → cần upgrade.

### 2.9 Cost spike & anomaly — phát hiện bất thường

| Metric | Định nghĩa | Lumi |
|---|---|---|
| **Daily spend vs 7-day avg** | today / avg_7d | ❌ |
| **Daily spend vs 30-day avg** | today / avg_30d | ❌ |
| **Threshold breach % (alert: >150% of avg)** | spike detection | ❌ |
| **Rolling z-score** | (today − μ_30d) / σ_30d, flag >3σ | ⚠️ có threshold median × 2, chưa z-score |
| **Per-user spike (today vs their 30d median)** | Catch user-level abuse | ❌ |

> **Ý nghĩa kinh tế:**
> - **150% of avg**: chuẩn industry. Nếu hôm nay tốn gấp 1.5× trung bình → cảnh báo.
> - **Z-score >3σ**: chuẩn statistical. >3 sigma = bất thường 99.7%.
> - **Per-user spike** phát hiện 1 user tự nhiên dùng gấp 10× thường ngày — có thể credentials bị leak.

## 3. "5 câu hỏi" của 1 cost tracking dashboard

Dashboard cost tốt phải trả lời 5 câu trong ≤ 5 giây:

| # | Câu hỏi | Metric trả lời | Lumi đáp ứng |
|---|---|---|---|
| 1 | **Tháng này đang tốn bao nhiêu?** | Cumulative cost, % budget | ⚠️ có cost period, chưa có budget |
| 2 | **Có vượt budget không?** | Projected EOM, days until exhausted | ❌ |
| 3 | **Ai đang tốn nhất?** | Top users ranking | ✅ |
| 4 | **Model nào tốn nhất?** | Cost per model breakdown | ✅ |
| 5 | **Có bất thường gì không?** | Spike alerts, z-score | ⚠️ có threshold compute, chưa surface UI |

→ Lumi đáp ứng **2/5 câu** rõ ràng. Build budget + alert UI là đủ chuẩn.

## 5. Ý nghĩa kinh tế từng metric — content cho `/docs` page

Content này dành cho người **không hiểu kỹ thuật** (BA, Marketing, CFO).

| Metric | Người không tech hiểu là gì | Câu chuyện kinh tế |
|---|---|---|
| **Total tokens** | Tổng "chữ" model đọc + viết | Volume base. Nhiều = tốn nhưng cũng có thể = nhiều giá trị tạo ra. So với revenue/output. |
| **Input vs output tokens** | Câu hỏi user vs câu trả lời model | Output đắt 3-30× input. Output >> input → model "lan man" → prompt engineering giảm 50% cost. |
| **Total cost** | Tổng tiền API Anthropic/OpenAI thu | $ damage thuần. So với budget. |
| **Cost per request** | Tiền 1 lượt gọi trung bình | Unit economics baseline. Nếu $0.05/req × 10K req/tháng = $500/tháng. |
| **Cost per session** | Tiền 1 cuộc trò chuyện | Critical. $0.50/session, user trả $5/tháng cho 10 session → biên 90%. Nếu cost/session > revenue/session → đang lỗ. |
| **Cost per user** | Tiền tiêu cho 1 user trong period | So với revenue/user. User free tốn $20 → đốt $20. |
| **Cost per 1K tokens** | "Đơn giá" thực tế (sau khi blend nhiều model) | Normalized comparison. Tháng trước $0.003/1K, tháng này $0.005/1K → đang dùng model đắt hơn. |
| **Cache hit ratio** | % câu "đã hỏi → rẻ hơn 90%" | Anthropic typical 50-85%. Càng cao càng tiết kiệm. |
| **Cache saving USD** | Tiền tiết kiệm nhờ cache | ROI của prompt engineering. Báo cáo CFO: "nhờ caching, tiết kiệm $X tháng này". |
| **Write amortization** | Mỗi $1 trả cho write, nhận lại bao nhiêu | 8.65× = mỗi $1 cache write → $8.65 saving sau đó. >3× là tốt. |
| **Cost per 1K requests** | Tiền cho 1000 lượt gọi | Hệ thống "đang đắt lên hay rẻ đi" theo time. |
| **Median cost per user** | "Tiền 1 user trung bình tiêu" | Robust to outlier. Không bị skew bởi 1 bot. |
| **p95 cost per user** | "Tiền của user thứ 95/100" | Heavy hitters threshold. |
| **TPM (tokens/min)** | "Tốc độ" hệ thống đang chạy | Capacity headroom. Peak TPM 80% rate limit → upgrade quota. |
| **Daily vs 7d avg %** | "Hôm nay tốn nhiều hơn TB tuần bao nhiêu" | >150% = cảnh báo. >200% = action ngay. |
| **Projected EOM** | "Nếu cứ tốc độ này, cuối tháng tốn $X" | Forecast budget breach trước khi xảy ra. |
| **Cost concentration top 10%** | "Top 10% user gây bao nhiêu % cost" | >80% = "long tail" — cân nhắc tier hoá user. |
